# Admin Feature Gap Assessment

## Scope and method

This assessment covers the discoverable admin-facing surfaces in:

- `apps/admin-console/app/admin/**`
- legacy admin routes under `apps/admin-console/app/platform/**`, `apps/admin-console/app/platform-admin/**`, and `apps/admin-console/app/tenant-admin/**`
- shared admin components under `apps/admin-console/components/**`
- supporting admin API routes and services under `apps/api/src/routes/**` and `apps/api/src/services/**`

Status legend used below:

- Fully built and wired
- Built but partial
- Placeholder only
- Legacy/orphaned
- Missing but recommended

## Top critical findings

1. The canonical admin nav still exposes several placeholder-only pages (`SSO / Identity`, `System Alerts`, `Tenant SSO`, `Tenant Roles`, `Permission Matrix`, `Session Management`, and others). These are not production-ready.
2. The admin app still has multiple parallel route trees (`/admin`, `/platform`, `/platform-admin`, `/tenant-admin`) that duplicate functionality and create operational ambiguity.
3. Tenant provisioning, tenant configuration, user administration, and role administration are the most operationally important real workspaces, but all remain partial and need stronger validation, workflow boundaries, and follow-through.
4. Several observability screens are either placeholder-only or only loosely connected to true operational data, which is risky ahead of an AWS migration where environment and integration visibility become more important.
5. The legacy tenant document admin flow uses direct user impersonation-style headers and should not be treated as production support tooling without hardening.
